## Executive summary

The HappyHero analytics subsystem repurposes a single shared OLTP PostgreSQL as both the application's source of truth *and* the analytics store. Behavioural events are not first-class — they are side-effect rows of normal operations, inferred at query time (moments from biometric timestamps, emotional moments from chat content, scores from per-question answers, behavioural Markov-state transitions from event-code tokenization). The pattern is reasonable at current scale (~1k events/sec) and the team has clearly invested in real analytics tooling (cohort segmentation bundle, Google Sheets exporters, forward-survival analysis, **per-segment week-over-week Markov-bigram comparisons via `cohort_markov_pivot.py`**).

Beneath that surface, the evaluation surfaced **18 findings (8 R-high, plus a confirmed high-impact hypothesis)** clustering into **four risk themes**, listed below in priority order. The dominant theme — **silent schema-semantics drift** — has already materialised twice in production (the 2026-05-11 `bot.questions` content-corruption incident, and the documented SQL-side H-collapse drift worked around in `cohort_markov_pivot.py`). Two themes (**T1, T4**) directly threaten the business drivers for which the subsystem exists; one (**T2**) is the predictable failure mode under growth; one (**T3**) is the compliance exposure given that the data is biometric/health.

The recommendations are conventional engineering moves (disable TypeORM auto-sync, add a read replica, separate DB users, propagate `app_version` to events) — none requires novel infrastructure. The non-obvious finding is **F16R**: the team *does* have a real week-over-week comparison framework (per `cohort_markov_pivot.py`), but the system cannot **attribute** an observed delta to a specific release. So BD2 is *coarsely* met, not fully met.

Single Postgres user (`pgvadmin`) across all services. Same DB serves OLTP and OLAP. Bot uses two `psycopg_pool` instances (sync + async), max_size default 4 each. NestJS uses TypeORM with poolSize 20, `synchronize: true` by default.

## 4. Architectural approaches identified (P4)

- **Modular monolith** at the DB level (single Postgres, schemas for organization)
- **Behavioural events as DB side-effect rows** (no separate event store)
- **Inferred events** at query time from row timestamps (moments, emotional moments, daily-check answer scoring)
- **Analytics-as-batch-export** to Google Sheets (gspread, thread-pool offloading per `bot-analytics-exporter` pattern)
- **Cohort segmentation as parameterised SQL** with shared exclusion list and per-language wrapper
- **Single shared DB user** + `SET search_path` for namespacing (Directus-era convention)
- **No circuit breaker, no shared retry primitive, no API versioning, no event-versioning**

### Confirmed hypothesis

**H1** — *"Inferred-event semantics have hidden coupling to the physical schema; routine column rename would silently change event counts; no integrity check."*

→ **Confirmed** via the 2026-05-11 `bot.questions` content-corruption incident, documented in `.claude/memory_bank/reports/test-content-audits-summary-2026-05-12.md`. Until 2026-05-11 17:05 UTC, `content_en` (HEXACO) and `content_en`/`content_uk` (PVQ) carried corrupted content; stored `user_scale_values` rows from before then for affected (test, language) pairs are computed against non-canonical items and are **not valid scores**. Workaround is a manual `--score-min-date '2026-05-11 17:05:00+00'` filter baked into the cohort wrapper — a code convention, not an automated integrity check. F9 (`synchronize: true`) is the underlying enabler.

## 7. Risk themes (P9)

### T1 — Silent schema-semantics drift (HIGH, rank 1)

**Member findings:** F9, F14R, F5, F17, F18R (and H1 as confirmed evidence).

Two distinct sub-clusters within this theme:

- **Cross-layer interfaces without contract tests** (F14R, F18R): score-domain knowledge spans Python compute → NestJS storage → SQL band thresholds; event-code taxonomy spans SQL collapse → shell legend → Python regex tokenizer → label map. These layers cannot be consolidated to one file — each is layer-appropriate. What's missing is a CI contract test asserting the cross-layer invariant. F14R is high-severity (the 2026-05-11 H1 incident is concrete precedent for this interface failing on score data); F18R is low-severity (event-code taxonomy is low-churn and the team's documented-workaround pattern is effective at observed drift cadence).

- **Intra-layer duplication** (F5): the 30-day cohort CTE is true cut-and-paste across 4 SQL files. Different remedy (consolidate, don't add contract test).

- **The architectural enabler** (F9): TypeORM `synchronize: true` is the upstream mechanism that allows source-data changes to ship without a review gate. Closing F9 reduces the surface that contract tests need to cover.

- **The change-concentration amplifier** (F17): three bot god-modules accumulate >50% of change traffic, hiding cross-layer touches inside high-velocity files.

The team has already lived through two instances: the 2026-05-11 `bot.questions` corruption (H1, confirming the score-domain interface failure), and the H-collapse SQL drift worked around in the Markov tokenizer (a smaller event-code interface failure). Severity of the theme is high *because of F9+F14R*, not because every site is equally fragile.

**Threatens:** BD1, BD2 (the entire purpose of the analytics subsystem).

### T2 — Compound failure under growth (HIGH, rank 2)

**Member findings:** F2 (TP), F6, F7, F8, F11, F12, F4R.

Today's small scale (1k eps) hides several interacting weaknesses. Together at 10× scale + a DB slowdown, they produce a textbook compound outage with corrupted analytics on the way out: shared OLTP for reads/writes, tiny bot pool, no circuit breaker, no backpressure, per-module retries with no shared budget, idempotency TODOs in three modules, Momenter→NestJS fan-out with no retry policy. Each is tolerable in isolation; together they aren't.

**Threatens:** BD1, BD2 (when the system can't capture events reliably under load, no analytics).

### T3 — PII/biometric audit boundary is structurally weak (HIGH, rank 3)

**Member findings:** F1R, F13.

Two independent gaps compound: a single shared `REPORTING_AUTH_TOKEN` with no per-analyst identity, *and* a single DB user `pgvadmin` across all services with no privilege separation. The system handles HR/HRV biometric data and chat content — regulator-attracting categories. The S7 exploratory scenario (subject access request from a regulator) is not just hard to answer — it's structurally unanswerable today.

**Threatens:** BD1, BD2 (regulatory action would shut analytics access down entirely; compliance preempts release-impact analysis).

### T4 — Release-impact attribution is partial (MEDIUM-HIGH, rank 4)

**Member findings:** F16R2, F10.

BD2 is "measure impact of app changes." The current state is *partial* attribution, not absent attribution. The team has `cohort_markov_pivot.py` producing week-over-week behavioural deltas, which answers "did anything change". What's missing is the ability to attribute observed deltas to *specific releases*:

- **Immediate binding constraint:** there's no `releases` table in shared-db. Adding one (with rollout-start and rollout-complete timestamps) plus teaching `cohort_markov_pivot.py` to join against it would unblock coarse attribution for fast/forced rollouts in 1-2 dev-days (R4a).
- **Residual constraint:** events carry no `app_version`, so during staged-rollout windows the timestamp-based bucket is approximate. Fully resolving this needs (a) a mobile-side `X-Client-Version` header, (b) `app_version` columns on session-anchoring tables, (c) R1.1 to land first so migrations are safe (R4b).

**Threatens:** BD2. Severity was downgraded from HIGH to MEDIUM-HIGH after a critical-questions review noted that (i) timestamp-based attribution covers the common case of single-feature-per-release-cycle weeks; (ii) the confound problem (multiple changes in one week) is real but not measured; (iii) the mitigation has non-trivial cost not priced in F16R's original framing.

## 8. Recommendations (P9)

Listed in priority order, each addresses exactly one theme. Estimates are order-of-magnitude.

### R1 — Establish a schema-change review gate and a single source of analytics semantics (effort L, addresses T1)

Three sub-actions, ordered by effort-to-value:

1. **`SYNC_DB=0` in all non-dev environments** and adopt an explicit migrations workflow with PR-level review of every schema change. Closes the F9 enabler of H1 by itself.
2. **Extract the cohort-membership SQL into one materialised view or shared snippet** that the four cohort SQLs include. Couples cohort changes to one file.
3. **Add CI contract tests** for the three locations where duplicated taxonomies live: (a) score-semantics across Python strategies / NestJS entity / cohort band cutoffs; (b) event-code taxonomy across `cohort_activity_chain_export.sql` / `cohort_markov_pivot.py` tokenizer / `EVENT_NAMES` label map; (c) cohort-membership CTE after R1.2 consolidates it. Turns "silent drift" from a class of failure into a detectable class.

### R2 — Decouple user-facing write path from analytics-read path before the 10× scale event (effort L, addresses T2)

1. **Add a Postgres read replica.** Migrate `bot/src/analytics/*` and `database/query.sh` to it. Eliminates the F2 tradeoff coupling.
2. **Introduce a shared retry primitive** (jitter, max-attempts, deadline). Migrate the two existing retries (`mobile_app_server.py:1419`, `payment_events_processor.py:116`) onto it.
3. **Add idempotency keys to the three flagged INSERT paths** (the moment-message presentation, payment events, dialogue messages). Resolves the explicit TODOs.
4. **Add a simple pool-saturation fail-fast policy** on shared-db acquires (or a real circuit breaker if effort permits). Doesn't have to be sophisticated; has to stop unbounded blocking under degradation.

### R3 — Make biometric-data access attributable (effort L, addresses T3)

1. **Replace single `REPORTING_AUTH_TOKEN` with per-analyst tokens** (or short-lived signed credentials via GCP IAM). Log identity at `auth_service.py`'s boundary.
2. **Split `pgvadmin` into per-service Postgres roles with GRANT/REVOKE** expressing least privilege.
3. **Append-only audit log** for sensitive-data reads (separate write-once table or external SIEM). Resolves F1R's row-mutability concern.

Standard infrastructure for systems handling regulated health data; none of it is novel.

### R4a — Releases table + release-aware cohort_markov_pivot (effort S, addresses T4, do now)

The cheapest action with the highest immediate value. Three sub-actions, all completable in 1–2 dev-days:

1. **Create a `releases` table in shared-db**: `{version, channel, released_at, rollout_complete_at, notes}`. Seed manually from git tags + Slack history for the last 12 months.
2. **Add `--releases-csv` to `cohort_markov_pivot.py`** so the existing week-over-week bigram analysis can attribute deltas to a primary release for the window. Surface "mixed-attribution" weeks explicitly when rollouts span more than one comparison window.
3. **Document the convention** in `cohort-segmentation-bundle.md`: fast/forced rollouts get reliable attribution; staged rollouts >1 week get approximate attribution (use R4b for those).

Unblocks the common case of single-feature-per-release-cycle weeks immediately. Does *not* require R1.1 first because the migration is just a new table.

### R4b — `app_version` on session-anchoring tables (effort M, addresses T4, depends on R1.1)

The deeper fix, for staged-rollout windows where R4a's timestamp join is approximate.

1. **PREREQUISITE: R1.1** must land first (`SYNC_DB=0` + migrations workflow) so the schema changes below ship safely.
2. **Confirm or add the `X-Client-Version` header** on every mobile-app request to bot-api and mobile-app-back. Likely a small Flutter change if absent.
3. **Add nullable `app_version VARCHAR(20)` to session-anchoring tables** only — `dialogues`, `daily_check`, `moments`. *Not* on per-message rows; the version doesn't change mid-session, so session-level stamping is sufficient. Do not backfill — pre-cutover events are bucketed as "app_version unknown".
4. **Update `cohort_markov_pivot.py`** to prefer `events.app_version` over the timestamp-bucket from R4a when present.

After R4a+R4b, S10's response measure ("analyst can list what events would invalidate the comparison") becomes structurally achievable. Effort estimate **M** reflects the mobile-side header rollout, schema migrations on three high-write tables, and analytics-tooling update — none individually large, but they sequence.

## 9. Known gaps in this evaluation

- **Solo run.** No real stakeholders consulted; "votes" in P7 are evaluator-only.
- **Architect was proxied.** Phases 6 and 8 answers were grounded in code reading and the existing memory bank, not in conversation with the team. Some findings reflect the *visible* architecture; the team may know operational details (env-var overrides, prod-only config) not in the repo.
- **No measurements.** Pool acquisition times, query p95 latencies, and audit retention durations are described qualitatively. R2 and R4 would benefit from a one-week telemetry pass before execution.
- **Two controller bugs surfaced during the run** and were fixed mid-evaluation (`controller.py` coverage-sort, `selection.py` closure short-circuit). The earlier dropped tick (`sec.blast-radius` selected then discarded) remains as a `Plan + AdaptiveDecision` pair with no follow-up Question in the chain — intentional artefact of the bug, not a gap in the evaluation.
- **`stipo.synthetic` initial findings.** F1, F2-original, F3-original, F4-original were initially generated from assumptions before I had read the code. F1 and F4 were superseded via `supersedes` links; F2 and F3 stood up to code review unchanged. See `reports/knownbugs.md` Section A for context.
